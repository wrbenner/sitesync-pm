Three things blocking you — fix in this order:

1Fresh npm install (5 min)

The installed node_modules are built for a different CPU architecture (likely arm64 vs x86_64). Vite's Rolldown bundler can't load its native bindings. On your Mac, run:

cd ~/Desktop/sitesync-pm
rm -rf node_modules package-lock.json
npm install
npm run dev
2Create a .env file with Supabase credentials (2 min)

src/lib/supabase.ts has hardcoded fallback credentials pointing to https://hypxrmcppjfbtlwuoafc.supabase.co. But you need a user account on that Supabase instance to get past the auth wall. Either:

Option A — Create an account on the existing Supabase instance via the Signup page (/signup), then log in normally.

Option B — Enable the dev bypass. Create a .env file with ONLY this:

VITE_DEV_BYPASS=true

⚠️ Important: do NOT also set VITE_SUPABASE_URL or VITE_SUPABASE_ANON_KEY — the dev bypass code (src/lib/devBypass.ts) requires those to be unset. If you have a .env.example copied over, delete the Supabase vars from it.

3Auth guard blocks everything (the actual "pages don't work")

src/components/auth/ProtectedRoute.tsx (line 183) wraps every route. If useAuth() returns no user, it does <Navigate to="/login" />. Since there's no valid session, you see nothing but the login screen. Once you either log in or enable the dev bypass, all 50+ pages will render.

If pages render but show empty data, that's because the Supabase database tables may be empty or RLS policies may be blocking reads. You'll need to seed data or adjust RLS.

The new sitesync-pm app is a different stack (Vite/React/Supabase), so the code can't be copy-pasted directly, but there's a LOT of domain intelligence and patterns worth recycling.

Python + AWS SQS. Plan classification, edge detection (Roboflow), overlap generation (OpenCV), dimensional discrepancy analysis, scale extraction — all Gemini-powered. Contains the most valuable AI prompts in the entire codebase.

Part 5: Priority Roadmap

Strategy: Get sitesync-pm running first, then bring in the old codebase's best assets (AI prompts, annotation layer, data model patterns) to fill the gaps your strategy docs identified.

Phase 1: Get It Running (Today)

1Fix deps + auth (30 min)

rm -rf node_modules package-lock.json && npm install, create .env with VITE_DEV_BYPASS=true (no Supabase vars), run npm run dev. All 50+ pages should render.

2Verify Supabase connection + seed data (1 hr)

Once bypass works, switch to real Supabase. Create account via /signup, create an org and project. Pages will render with empty data. Run the existing migrations to ensure tables exist.

Phase 2: Recycle the Old App's Best Assets (1-2 weeks)

3Port the PDF Annotation Layer from drawing-tools (2-3 days)

The old drawing-tools repo has a clean Canvas-based PDFAnnotationLayer in Tailwind/React that supports rectangle + polygon + text tools, undo/redo, color picker, and scale-aware coordinates. Port it into sitesync-pm's DrawingViewer. Wire annotations to persist in the Supabase annotations table (migration already exists). This gives you the "Bluebeam killer" from the Demo Build Spec.

4Port the AI classification + discrepancy prompts (1-2 days)

The plan-classification-v2 microservice has a master Gemini prompt that extracts sheet number, drawing title, discipline, scale, floor level, building type, and more from construction drawings — with anti-hallucination rules built in. Port this into a Supabase edge function (classify-drawing already exists). The dimensional-discrepancy-analysis prompt is the other crown jewel — it compares arch vs struct drawings and identifies dimensional conflicts with severity ratings.

5Port the overlap comparison viewer (1 day)

drawing-tools has a clean two-layer PDF overlay tool with opacity/position/scale controls. Port to sitesync-pm's Drawings page. Wire it to the overlap-image-generation Python service if you want server-side overlap processing too.

6Translate old Prisma schema → Supabase migrations (2-3 days)

The old backend has 30+ Prisma models covering entities, annotations, drawing pairs, discrepancy reports, and billing that sitesync-pm doesn't fully have yet. Translate the key models into PostgreSQL migrations. Focus on: EntityInstance (annotations), ArchStructDrawingPair, DrawingPairDiscrepancyReport, ConvertedPageImage (with classification metadata).

Phase 3: Build the Demo Showstoppers (2-3 weeks)

7Complete RAG Project Brain (2-3 days)

Add pgvector extension, build document_chunks table, create embedding pipeline (on file upload → extract text → embed with Gemini text-embedding-004 → store vectors). Wire project-brain-query to do real vector similarity search. The ProjectBrain.tsx component already exists.

8Add resumable uploads with tus (1-2 days)

Install tus-js-client. Replace upload handlers in FileUpload, DrawingUpload, BudgetUpload. The old app's signed-URL upload with progress tracking is a reference pattern.

9Integrate OpenSign for e-signatures (2-3 days)

Deploy OpenSign via Docker. Build send-for-signature edge function. Add "Send for Signature" to Change Orders, Lien Waivers, Contracts pages.

10Complete Safety Photo AI loop (1 day)

The analyze-safety-photo edge function works. Wire: auto-create tasks from violations, persist safety scores, add OSHA reference lookup.

Wire up @orama/orama (already in deps) for Cmd+K search across all entities. Upgrade to Typesense later.

Email Notifications

Bottom Line

Your new app (sitesync-pm) is architecturally superior to the old one — it has 3x more construction workflows, state machines for every entity lifecycle, offline-first architecture, real-time collaboration, and enterprise auth/permissions. The old app had none of these.

What the old app HAS that the new one DOESN'T is the drawing intelligence pipeline — the AI prompts for plan classification, scale extraction, dimensional discrepancy analysis, and the PDF annotation/overlap tools. These represent months of domain-specific iteration and are the core IP of the product.

The path forward:

  1. Fix deps + auth today → pages work
  2. Port drawing intelligence from old codebase → unique differentiator
  3. Build RAG + e-signatures + resumable uploads → demo showstoppers
  4. Deploy old Python microservices alongside → full AI pipeline
