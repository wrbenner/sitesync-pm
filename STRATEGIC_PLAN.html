Phase 1 — Kill Every Fake Button

    Connect handleApprove → submittalService.addApproval(). Connect handleRequestRevision → submittalService.createRevision(). Wire document upload metadata persistence. The service layer is complete — just never imported.
    SubmittalDetail.tsx:48-61, submittalService.ts:204, :263
  • Pay Apps: Wire submit + mark-as-paid + process payment

    PaymentApplications.tsx:139, 153, 1038

Phase 2 — The Financial Backbone

Connect Contracts ↔ Change Orders ↔ Budget ↔ Pay Apps

4 sessions · ~$160–240 · This is what separates Procore from toys

Right now the financial modules are isolated islands. A change order approval should ripple through the budget. A pay app should bill against a contract's SOV. This phase makes them talk to each other — the way Swinerton expects.

  • Change Order line items table + CRUD
    Create change_order_line_items migration (cost_code, description, amount, scope_type). Replace single flat "amount" with itemized breakdown. Add/edit/remove line items in the CO detail view.
  • CO approval → budget auto-update
    When a change order status moves to "approved", automatically update budget_items committed amounts, project contract_value, and trigger EAC recalculation. This is the single most important financial connection.
  • Fix CO promotion logic
    Change usePromoteChangeOrder from creating a new record to mutating the original's type field (PCO → COR). Maintain audit trail of the conversion. Single source of truth.
  • Pay Apps: Link billing to Contract SOV line items
    Pay app line items should reference contract_sov_items. Track "completed to date" per SOV line. Calculate retention per line, not just globally. Show billing progress against each SOV item.
  • Budget: Dynamic contingency + CO-driven revisions
    Replace hardcoded $3.8M contingency with real contingency cost code. When COs approve, auto-revise budget and recalculate. Wire PO commitments to budget lines via procurement data.
  • Contract amendments workflow
    Add amendment creation to the Contracts page. Track amendment history, effective dates, impact on contract value. Link amendments to change orders that triggered them.
  • Entity linking: CO ↔ RFI ↔ Submittal ↔ Drawing
    Create entity_links junction table (source_type, source_id, target_type, target_id). Replace all hardcoded empty arrays with real cross-module relationships. Show linked items in every detail panel.
Phase 3 — Bulletproof Quality

Error Boundaries, E2E Tests, Email Delivery

3 sessions · ~$120–180 · Makes it trustworthy

A platform Swinerton trusts can't crash with a white screen when a network request fails. Every page needs error protection, critical workflows need automated tests, and notifications need to actually arrive.

  • Add ErrorBoundary to all 35+ unprotected pages
    Wrap every route-level component in the existing ErrorBoundary pattern. Show "Something went wrong" with retry button instead of white screen. Currently only Dashboard, RFIs, Budget, Schedule, Submittals are protected.
  • E2E tests for 5 critical workflows
    Playwright tests for: (1) RFI create → assign → respond → close, (2) Submittal create → approve, (3) Change order create → approve → budget update, (4) Pay app create → submit → approve, (5) Punch item create → complete → verify.
  • Wire email delivery via Resend
    processNotificationQueue() exists but is never called. Create a Supabase Edge Function that runs on a cron to process the queue and send emails via Resend API. RFI assignments, submittal reviews, overdue alerts — real emails.
  • RLS policies audit + enforcement
    Every table needs Row Level Security so users only see their project's data. Audit all tables, add project_id-scoped RLS policies, test that cross-project data is invisible.
  • Drawing linkage for RFIs + Punch Items
    Replace text-only drawing references with a real drawing picker component. Select from project drawings, show thumbnail preview. Store as FK relationship, not text string.

Phase 5 — Production Launch

Deploy, Monitor, and Polish

2 sessions + infrastructure setup · ~$80–120 in Claude + ongoing infra costs

Ship it for real. Custom domain, proper hosting, error monitoring, performance optimization. This is the last mile.

  • Deploy to Vercel with custom domain
    Set up sitesyncpm.com (or similar). Preview deployments for PR branches. Environment variables for Supabase keys. Edge caching for static assets.
  • Upgrade Supabase to Pro
    Daily backups, no pause after inactivity, point-in-time recovery, 8GB database, dedicated compute. The free tier pauses after 1 week of inactivity — unacceptable for a product.
  • Sentry error tracking
    Global Sentry.init in main.tsx (foundation already exists). Source maps upload on deploy. Slack alerts for new errors. Know when things break before users tell you.
  • Performance audit + optimization
    Lighthouse audit. Code-split heavy pages (BIM viewer, Drawing viewer). Lazy-load below-fold components. Optimize Supabase queries (add indexes for common filters). Target: LCP under 2s.
  • Clean up duplicate files
    Remove all "filename 2.tsx" duplicates throughout the codebase (ScheduleKPIs 2.tsx, index 2.tsx, PunchListDetail 2.tsx, etc.). These are macOS copy artifacts polluting the build.
