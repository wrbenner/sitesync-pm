## Overview

SiteSync uses a `fromTable<T>()` helper defined in `src/lib/supabase.ts` that wraps `supabase.from()` with proper TypeScript generics. This eliminates `as any` casts and gives full type safety on query results, inserts, and updates. Every Supabase query in the codebase must go through this helper — never call `supabase.from()` directly.

## Detection

Look for any of these patterns — they are all candidates for this skill:

```typescript
// BAD — explicit any cast
const { data } = await supabase.from('projects').select('*') as any;

// BAD — result assigned to loosely typed variable
const result = await supabase.from('projects').select('*');
const data = result.data as any;

// BAD — direct supabase.from() without fromTable helper
const { data, error } = await supabase.from('daily_logs').select('*');
```

Also scan for `// @ts-ignore` comments above Supabase calls — these usually hide an underlying type problem this skill resolves.

## The `fromTable<T>()` Helper

Location: `src/lib/supabase.ts`

```typescript
// Full pattern — copy this into src/lib/supabase.ts if it doesn't exist yet
import { createClient } from '@supabase/supabase-js';
import type { Database } from '../types/database.types';

export const supabase = createClient<Database>(
  import.meta.env.VITE_SUPABASE_URL,
  import.meta.env.VITE_SUPABASE_ANON_KEY
);

/**
 * Type-safe wrapper around supabase.from().
 * T must match a table name defined in Database['public']['Tables'].
 * Returns a fully-typed QueryBuilder — no `as any` needed downstream.
 */
export function fromTable<T extends keyof Database['public']['Tables']>(table: T) {
  return supabase.from(table);
}
```

The type parameter `T` is constrained to actual table names from the generated `Database` type. TypeScript will error at compile time if you pass a table name that doesn't exist.

## Resolution Steps

### Step 1 — Identify the table type

Check `src/types/database.types.ts` (auto-generated by Supabase CLI) for the correct row type:

```typescript
// In database.types.ts, a table looks like:
Database['public']['Tables']['projects']['Row']  // SELECT result type
Database['public']['Tables']['projects']['Insert']  // INSERT type
Database['public']['Tables']['projects']['Update']  // UPDATE type
```

### Step 2 — Replace the raw query

```typescript
// BEFORE
const { data, error } = await supabase.from('projects').select('*') as any;

// AFTER
import { fromTable } from '@/lib/supabase';

const { data, error } = await fromTable('projects').select('*');
// data is now typed as Database['public']['Tables']['projects']['Row'][] | null
```

### Step 3 — Handle specific column selections

When selecting specific columns, TypeScript narrows the return type automatically:

```typescript
// Selecting specific columns — still fully typed
const { data, error } = await fromTable('projects')
  .select('id, name, status, created_at');
// data is typed as Pick<ProjectRow, 'id' | 'name' | 'status' | 'created_at'>[] | null
```

### Step 4 — Typed inserts and updates

```typescript
import type { Database } from '@/types/database.types';

type ProjectInsert = Database['public']['Tables']['projects']['Insert'];
type ProjectUpdate = Database['public']['Tables']['projects']['Update'];

// INSERT — TypeScript enforces required fields
const newProject: ProjectInsert = {
  name: 'Site Alpha',
  status: 'active',
  org_id: orgId,
};
const { data, error } = await fromTable('projects').insert(newProject).select();

// UPDATE — all fields optional
const update: ProjectUpdate = { status: 'completed' };
const { data, error } = await fromTable('projects')
  .update(update)
  .eq('id', projectId)
  .select();
```

### Step 5 — Handle the error

Always check `error` before using `data`. Supabase never throws — it returns `{ data: null, error: PostgrestError }` on failure:

```typescript
const { data, error } = await fromTable('projects').select('*');
if (error) throw new Error(`Failed to fetch projects: ${error.message}`);
if (!data) return [];
// data is safely typed here
```

### Step 6 — Verify TypeScript compiles

```bash
npx tsc --noEmit
```

Zero errors expected. If you see `Type 'string' is not assignable to type 'never'` errors, the `Database` types may be stale — regenerate with:

```bash
npx supabase gen types typescript --project-id $PROJECT_ID > src/types/database.types.ts
```

## Common Pitfalls

| Pitfall | Symptom | Fix |
|---|---|---|
| Using `.select('*')` then casting result | `data as ProjectRow[]` | Remove cast — `fromTable` types it automatically |
| Stale generated types | TypeScript complains about valid columns | Regenerate with `supabase gen types` |
| Using raw `supabase.from()` in a new hook | Code works but loses type safety | Always import and use `fromTable` |
| Forgetting `.select()` after `.insert()` / `.update()` | `data` is `null` unexpectedly | Chain `.select()` to get the mutated row back |
| RLS blocking query silently | `data` is `[]`, no error | Check Supabase Dashboard → Auth → Policies |

## Construction App Context

SiteSync tables you'll encounter most often:

- `projects` — Top-level job sites
- `daily_logs` — Field superintendent daily logs
- `punch_items` — Punch list entries (defect tracking)
- `rfis` — Requests for information
- `submittals` — Submittal tracking
- `users` / `org_members` — User/role management

All follow the same `fromTable<T>()` pattern.
