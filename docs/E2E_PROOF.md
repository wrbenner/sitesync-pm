# 90-Day Lifecycle E2E Proof

**The forever-regression-cliff.** A single Playwright spec that walks the full
90-day construction lifecycle on a deterministic seeded project, asserts every
load-bearing seam, and captures evidence to disk. When this spec is green, the
platform is demo-ready. When it's red, the SUMMARY.md row names the seam that
broke.

This is **not** a unit test, an integration test, or a CI commit-gate. It is
the *demo gate* — the slow, exhaustive proof we run before showing the platform
to a stakeholder, an investor, or a real GC.

---

## What this spec proves

Twelve labeled steps mirror the construction lifecycle from kickoff through
closeout. Each step asserts a real seam — not a UI lookalike — so a green run
means the chain actually fires, the math actually balances, and the artifacts
actually exist on disk.

| # | Step | What's asserted |
|---|------|------------------|
| 00 | Seed 90 days into the fixture project | Seed script exits 0, writes ≥100 rows |
| 01 | Login + crews on site | Most-recent workday has ~6 `crew_checkins` |
| 02 | Photo capture → drawing/crew/log linkage | ≥100 `photo_pins` with realistic GPS distribution |
| 03 | RFI ball-in-court routing | All `open` RFIs sit with the architect |
| 04 | Overdue sweep fires → follow-up | `drafted_actions` of type `rfi_overdue_sweep` exist with valid payload |
| 05 | Submittal rejected → RFI auto-drafted | `drafted_actions` of type `submittal_rejected` exist |
| 06 | Daily-log auto-draft assembled | ≥60 `daily_logs`, ≥1 submitted, with `daily_log_entries` |
| 07 | Pay app #4 missing waivers → block | Pay app #4 has zero `lien_waivers` (audit gap reproducible) |
| 08 | Pay app #3 G702/G703 reconciles | Lines 3, 6, 9 balance to the penny |
| 09 | Change order from RFI auto-CO | ≥1 `change_order` with `metadata.source = 'rfi'` |
| 10 | Punch closed with photo | ≥1 `verified` punch with both `before` and `after` photos |
| 11 | Audit-chain hash verifier passes | `verifyAuditChain` reports zero gaps |
| 12 | Compliance pack PDF exportable | `wh347` + `sealedExport` modules load + expose entry points |

Steps 11 and 12 SKIP cleanly when their dependencies (audit log rows / sealed
export edge function) aren't present on a given branch — they are deferral-aware,
not failure-prone.

## How to run

The spec depends on a real Supabase backing the fixture and a running dev
server. Run order matters:

```bash
# 1. Make sure local Supabase is up and migrations are applied.
supabase start
supabase db reset   # one-time, or whenever schema diverges

# 2. Set the env the seed + spec need.
export SUPABASE_URL="http://127.0.0.1:54321"
export SUPABASE_SERVICE_ROLE_KEY="$(supabase status --output json | jq -r .service_role_key)"

# 3. (Optional) Run the seed standalone first to validate it before
#    invoking Playwright. The spec invokes it again as Step 00, so this
#    is purely diagnostic.
bun run scripts/seed-90-day-lifecycle.ts

# 4. Run the spec. --headed gives you a browser window for visual sanity.
bunx playwright test e2e/lifecycle/90-day-smoke.spec.ts --headed

# 5. (Optional) Re-render SUMMARY.md after a crash.
npx tsx scripts/lifecycle-summary.ts
```

The spec's `webServer` block in `playwright.config.ts` already starts the Vite
dev server with `VITE_DEV_BYPASS=true`, so the spec can navigate to routes
without an auth flow.

## Where the evidence lives

Everything goes to `audit/lifecycle-proof/`:

```
audit/lifecycle-proof/
├── SUMMARY.md                  ← human-readable proof report
├── step-00.json … step-12.json ← per-step structured evidence
├── step-00.png … step-12.png   ← visual evidence (screenshots)
└── console-errors.json         ← page console errors captured during the run (forensic)
```

`SUMMARY.md` carries one row per step with status (PASS / FAIL / SKIP), an
evidence link to the JSON + screenshot, and a short note. The JSON files
carry the raw measurement that drove the status — counts, distributions,
returned error messages — so a failure can be diagnosed without re-running
the spec.

## What "green" means

A green run satisfies all three of:

1. **All 12 steps PASS or have a documented SKIP.** No FAIL rows.
2. **Audit-chain verifier returns zero gaps** (or SKIP if the project has
   no audit-log rows yet — the verifier's contract is "no evidence of
   tampering," not "evidence of immutability").
3. **`SUMMARY.md` exists** and was generated by either the spec's own
   `finally` block or `lifecycleReporter` / `scripts/lifecycle-summary.ts`.

Anything less is not green. A spec that "ran fast and crashed" is not the same
as a spec that "ran slow and proved 12 things."

## Failure-mode diagnosis

When a step fails, look here first. The right-hand column tells you which file
to read or which seam to inspect.

| If step ___ FAILS | The seam is | Where to look |
|---|---|---|
| 00 (seed) | seed script vs schema | `scripts/seed-90-day-lifecycle.ts` — read the stderr in the step JSON; usually a column-name drift after a migration |
| 01 (crews on site) | seed didn't write `crew_checkins` for recent workdays | `scripts/seed-90-day-lifecycle.ts:seedCheckins` — check the WORKDAYS array and `checked_in_at` column |
| 02 (photo linkage) | `photo_pins` table or its `gps_status` column | `supabase/migrations` for the latest photo-pin migration |
| 03 (RFI routing) | `rfis.ball_in_court` not being set on insert | seed `seedRfis` — the `ball_in_court: USERS.architect` line for `open` status |
| 04 (overdue sweep) | `drafted_actions` table missing OR no `rfi_overdue_sweep` rows | `src/lib/crossFeatureWorkflows.ts` for the chain; seed's `seedDraftedActions` for the fixture |
| 05 (submittal → RFI) | same as 04 but for `submittal_rejected` chain | `src/lib/crossFeatureWorkflows.ts` |
| 06 (daily log) | `daily_logs` / `daily_log_entries` rows missing or wrong status | `seedDailyLogs` — and remember the column name is `log_date`, not `date` |
| 07 (pay app waivers) | pay app #4 was given waivers (the audit-gap signal disappeared) | `seedPayApps` — the `if (pa.status === 'draft') continue` guard is what produces the gap |
| 08 (G702 math) | line-item math drifted on a recent migration | inspect the JSON evidence; the failure note names the exact line that didn't balance |
| 09 (auto-CO) | `change_orders.metadata.source` not being set | `seedChangeOrders` — the last row sets `metadata: { source: 'rfi', ... }` |
| 10 (punch photos) | `punch_items.photos` is no longer JSONB or schema changed shape | `seedPunchItems` — the `photos` array of `{ caption: 'before' \| 'after' }` |
| 11 (hash chain) | `audit_log` rows exist but verifier reports gaps | `src/lib/audit/hashChainVerifier.ts` — the trigger formula must match exactly |
| 12 (compliance pack) | `src/lib/compliance/wh347` or `src/lib/audit/sealedExport` import threw | check the step note; usually a path rename after a refactor |

A SKIP that should have been a PASS is also a regression — the step note tells
you which dependency was missing.

## Design intent

A few choices that won't be obvious from reading the code:

- **One big test() with test.step()** — not 12 individual tests. We want a
  failure mid-lifecycle to halt downstream steps so a single root cause
  doesn't generate 11 cascading failures.
- **`safeStep` swallows assertion errors and re-throws after recording.**
  The recording is the load-bearing artifact; the throw exists so Playwright
  shows a red status. Without `safeStep`, a thrown assertion would crash
  before SUMMARY.md flushed.
- **DB assertions, not UI assertions, are load-bearing.** Screenshots are
  evidence — useful for visual regression review — but if a chain fires and
  writes the right row, that's what matters. UI flake is a separate concern.
- **Service-role Supabase for reads.** RLS is the wrong scope for a smoke
  test we own end-to-end. The `serviceClient` helper is e2e-only and is
  documented as such; don't use it from app code.
- **Steps 11 + 12 are SKIP-aware on purpose.** Hash-chain rows aren't
  populated by the seed (the chain trigger fires on real writes from edge
  functions), and the sealed-export edge function isn't deployed on every
  branch. SKIP keeps the run green when those features are intentionally
  deferred — the SUMMARY note tells you which.

## When this spec changes

The spec is meant to be stable. Edits should be rare. Edit when:

- A new lifecycle phase ships (e.g., a procurement loop) → add a step.
- A claimed cross-feature chain ships (e.g., `chain:weather_delay`) → add
  an assertion that the chain produces a `drafted_actions` row.
- The schema column names drift → update the corresponding seed function
  AND the spec's read query in lockstep.

Do **not** edit the spec to make a failing step pass. The spec catching a
regression is the point.
