# Compliance Officer Guide

You own the controls that prevent embarrassing rejections — pay apps that come back missing waivers, COIs that lapsed mid-period, prevailing-wage filings that drift from federal minimums. SiteSync surfaces every one of these as a deterministic check rather than a manual to-do.

## COI lifecycle

Insurance certificates are gated by [src/lib/coi/expirationGate.ts](../../src/lib/coi/expirationGate.ts). The cron watcher is [supabase/functions/coi-expiration-watcher/index.ts](../../supabase/functions/coi-expiration-watcher/index.ts); it runs daily at 06:00 UTC (cron registration is pending per [STATUS.md](../STATUS.md)) and:

- Sends reminders at T-14d, T-7d, T-3d, T-1d
- At T-0, inserts a `coi_check_in_blocks` row per [supabase/migrations/20260429020001_coi_check_in_block.sql](../../supabase/migrations/20260429020001_coi_check_in_block.sql)
- Always logs every reminder to `coi_expiration_alerts` even if the email channel is unavailable

The block clears the moment a new verified cert is uploaded.

## Lien waivers

Five templates ship in [src/lib/lienWaiver/templates](../../src/lib/lienWaiver/templates):

- AIA G706 conditional progress (default fallback)
- AIA G706 unconditional progress
- California (Cal. Civ. Code § 8132)
- Texas (Tex. Prop. Code § 53.281)
- Florida (Fla. Stat. § 713.20)

The renderer is [src/lib/lienWaiver/templateRenderer.ts](../../src/lib/lienWaiver/templateRenderer.ts); the PDF generator is [supabase/functions/lien-waiver-generator/index.ts](../../supabase/functions/lien-waiver-generator/index.ts).

### Tamper-detection contract

Per [docs/PLATINUM_FINANCIAL.md](../PLATINUM_FINANCIAL.md), every signed waiver row carries `content_hash` (SHA-256 of the rendered body at signing time) and `signed_body` (the verbatim text). At audit time, recompute the hash; drift implies tampering. The migration is [supabase/migrations/20260501120001_lien_waiver_signatures.sql](../../supabase/migrations/20260501120001_lien_waiver_signatures.sql).

### Legal review caveat

All five template bodies contain a `[TODO_LEGAL_REVIEW]` placeholder. The mechanical fields (sub name, project, period, amount, signature block) are correct. The prose is not. Counsel must approve verbatim statutory text before any waiver is exchanged with subs in production. See the wiring backlog in [STATUS.md](../STATUS.md).

## Pay-app pre-submission audit

The five checks in [src/pages/payment-applications/auditChecks.ts](../../src/pages/payment-applications/auditChecks.ts) are deterministic:

| ID | Trigger |
| --- | --- |
| `lien_waivers_present` | Sub billed work this period but has no non-pending waiver |
| `coi_active_for_period` | Sub billed work but has no verified COI covering the period |
| `g702_g703_reconcile` | G703 line totals do not equal G702 header total (>$1 drift) |
| `sov_percent_under_100` | Any SOV line bills > 100% of scheduled value |
| `retainage_math_correct` | Retainage amount is inconsistent with totals × percent |

Submit is disabled until all five pass — or the PM accepts the gaps with a ≥ 12-character reason, written to `payapp_audit_overrides` per [supabase/migrations/20260429020000_payapp_audit_overrides.sql](../../supabase/migrations/20260429020000_payapp_audit_overrides.sql).

The server-side mirror is [supabase/functions/payapp-audit/index.ts](../../supabase/functions/payapp-audit/index.ts) — that's the authoritative gate. Even if the UI is bypassed, the edge function rejects unless the override is present.

## Sub identity caveat

Per [docs/COMPLIANCE_GATE.md](../COMPLIANCE_GATE.md), sub identity is currently name-based, not ID-based, in `lien_waivers`. Matching uses case-insensitive substring comparison. Watch for typos — a sub named "ACME Drywall" and "Acme Drywall, Inc." will match, but "ACME Drywell" will not. A future migration will add `subcontractor_id uuid`.

## Prevailing wage (Davis-Bacon)

The pure decision engine is [src/lib/compliance/prevailingWage](../../src/lib/compliance/prevailingWage). Decisions are persisted per [supabase/migrations/20260502110000_prevailing_wage_decisions.sql](../../supabase/migrations/20260502110000_prevailing_wage_decisions.sql) and synced from federal sources via [supabase/functions/prevailing-wage-sync/index.ts](../../supabase/functions/prevailing-wage-sync/index.ts).

Caveat from [docs/ENTERPRISE_COMPLIANCE_PACK.md](../ENTERPRISE_COMPLIANCE_PACK.md): the engine takes per-worker fringe inputs directly; it does not yet pull from the customer's benefit-plan registry. Wire that data when its shape is stable.
